The `climate:` component models HVAC systems, thermostats, and other temperature-control devices. It exposes rich state to Home Assistant â€” operating mode, fan mode, swing mode, preset, current temperature, target temperature, and humidity â€” and accepts commands through the `climate.control` action. This page covers the full shared configuration, all mode enumerations, automation triggers, the control action, and examples of the most popular platform implementations.

## Base Configuration

```yaml
climate:
  - platform: <platform_name>
    name: "Living Room AC"      # Entity name exposed to HA / MQTT
    id: living_room_ac          # C++ variable ID for automations
    icon: "mdi:air-conditioner" # MDI icon string
    entity_category: ""         # "" | diagnostic | config
    visual:                     # Override HA UI display range
      min_temperature: 16 Â°C
      max_temperature: 30 Â°C
      temperature_step: 0.5 Â°C  # or separate target/current steps
      min_humidity: 30
      max_humidity: 80
    on_state: ...               # Automation triggers
    on_control: ...
```

### Visual Overrides

The `visual:` block customises the temperature slider and display in the Home Assistant frontend without affecting the hardware limits. `temperature_step` can be a single value applied to both target and current temperature, or separate values:

```yaml
visual:
  min_temperature: 10
  max_temperature: 35
  temperature_step:
    target_temperature: 0.5
    current_temperature: 0.1
```

## Climate Modes

Climate modes control the overall operating mode of the device.

| Value | C++ Enum | Description |
|---|---|---|
| `OFF` | `CLIMATE_MODE_OFF` | System is off |
| `HEAT_COOL` | `CLIMATE_MODE_HEAT_COOL` | Automatic heat/cool (range setpoint) |
| `COOL` | `CLIMATE_MODE_COOL` | Cooling only |
| `HEAT` | `CLIMATE_MODE_HEAT` | Heating only |
| `DRY` | `CLIMATE_MODE_DRY` | Dehumidification |
| `FAN_ONLY` | `CLIMATE_MODE_FAN_ONLY` | Ventilation without heating/cooling |
| `AUTO` | `CLIMATE_MODE_AUTO` | Platform-specific auto mode |

## Fan Modes

Fan speed and pattern for platforms that expose fan control.

| Value | C++ Enum | Description |
|---|---|---|
| `ON` | `CLIMATE_FAN_ON` | Fan always on |
| `OFF` | `CLIMATE_FAN_OFF` | Fan off |
| `AUTO` | `CLIMATE_FAN_AUTO` | Automatic fan speed |
| `LOW` | `CLIMATE_FAN_LOW` | Low speed |
| `MEDIUM` | `CLIMATE_FAN_MEDIUM` | Medium speed |
| `HIGH` | `CLIMATE_FAN_HIGH` | High speed |
| `MIDDLE` | `CLIMATE_FAN_MIDDLE` | Middle speed |
| `FOCUS` | `CLIMATE_FAN_FOCUS` | Focused / concentrated flow |
| `DIFFUSE` | `CLIMATE_FAN_DIFFUSE` | Diffused / wide flow |
| `QUIET` | `CLIMATE_FAN_QUIET` | Quiet / silent mode |

Platforms can also expose custom fan modes as strings via `custom_fan_mode`.

## Automation Triggers

<CardGroup cols={2}>
  <Card title="on_state" icon="circle-dot">
    Fires whenever the climate entity state changes (mode, temperature, fan mode, etc.).
    Variable `x` is a reference to the `Climate` object.
  </Card>
  <Card title="on_control" icon="sliders">
    Fires when a `climate.control` action is received, **before** the platform applies it.
    Variable `x` is a reference to the `ClimateCall` object containing the requested changes.
  </Card>
</CardGroup>

```yaml
climate:
  - platform: bang_bang
    ...
    on_state:
      then:
        - logger.log:
            format: "Climate mode changed to %d"
            args: ["(int) x.mode"]
    on_control:
      then:
        - logger.log: "Climate control received"
```

## climate.control Action

Use `climate.control` to programmatically change any aspect of the climate entity from automations.

```yaml
action:
  - climate.control:
      id: living_room_ac
      mode: COOL
      target_temperature: 22.0
      fan_mode: AUTO
      swing_mode: VERTICAL
      preset: ECO
```

<Accordion title="climate.control parameters">

| Key | Type | Description |
|---|---|---|
| `mode` | ClimateMode | Operating mode (OFF, HEAT, COOL, etc.) |
| `target_temperature` | float (Â°C) | Single target temperature |
| `target_temperature_low` | float (Â°C) | Lower bound for HEAT_COOL mode |
| `target_temperature_high` | float (Â°C) | Upper bound for HEAT_COOL mode |
| `target_humidity` | integer (%) | Target humidity percentage |
| `fan_mode` | ClimateFanMode | Standard fan mode |
| `custom_fan_mode` | string | Platform-specific fan mode string |
| `preset` | ClimatePreset | Standard preset |
| `custom_preset` | string | Platform-specific preset string |
| `swing_mode` | ClimateSwingMode | Swing direction |

</Accordion>

All values support templating via `!lambda`:

```yaml
action:
  - climate.control:
      id: thermostat
      target_temperature: !lambda "return id(schedule_temp).state;"
```

```yaml

  - platform: bang_bang
```

<Tip>
When using `bang_bang` or `thermostat`, the `sensor:` key must reference the `id` of a `sensor:` component already defined in your configuration. The sensor's `update_interval` directly affects how quickly the thermostat responds to temperature changes.
</Tip>
