ESPHome's secrets system lets you store sensitive values — Wi-Fi credentials, API encryption keys, OTA passwords — in a separate `secrets.yaml` file that lives outside your device configs. Device YAML files reference those values with the `!secret` tag, and the actual strings are injected at build time without ever appearing in the main configuration. This page explains the file format, lookup rules, what to store there, and how to keep it safe.

---

## The `secrets.yaml` file

`secrets.yaml` is a flat YAML key/value file. Each key is a secret name; each value is the secret's content. The file uses standard YAML, so strings, integers, and booleans are all valid values.

```yaml
# secrets.yaml
wifi_ssid: "My Home Network"
wifi_password: "correct-horse-battery-staple"
api_key: "dGhpcyBpcyBhIHRlc3Qga2V5IGZvciBkb2Nz"
ota_password: "my-super-secret-ota-pw"
fallback_ap_password: "fallback-access-pw"
mqtt_username: homeassistant
mqtt_password: "mqtt-broker-password"
```

<Note>
ESPHome recognises both `secrets.yaml` and `secrets.yml` as the secrets filename (from `SECRETS_FILES` in `esphome/const.py`). The `.yaml` form is conventional.
</Note>

---

## The `!secret` tag

Reference a secret anywhere in a device config file using the `!secret` YAML tag followed by the key name:

```yaml
wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password

api:
  encryption:
    key: !secret api_key

ota:
  - platform: esphome
    password: !secret ota_password
```

At build time ESPHome replaces each `!secret key` with the corresponding value from `secrets.yaml`. If the key is not found, the build fails with a clear error pointing to the location in your config.

---

## Secrets file lookup order

ESPHome's YAML loader implements a two-step lookup when resolving `!secret` references (from `yaml_util.py`):

<Steps>
  <Step title="Same directory as the config file">
    The loader first looks for `secrets.yaml` in the same directory as the YAML file that contains the `!secret` tag. This allows per-project or per-directory secrets.
  </Step>
  <Step title="Main config directory (fallback)">
    If the referencing file is not the top-level config file (e.g. it is an `!include`d package) and the first lookup fails, ESPHome falls back to `secrets.yaml` in the same directory as the top-level config file.
  </Step>
</Steps>

This means you can safely use `!secret` inside package files — as long as the corresponding key exists in `secrets.yaml` next to the device's main YAML, the value will be found.

## What to store in `secrets.yaml`

<CardGroup cols={2}>
  <Card title="Wi-Fi credentials" icon="wifi">
    `wifi_ssid`, `wifi_password`, and fallback AP password. These are the most common secrets.
  </Card>
  <Card title="API encryption key" icon="key">
    The 32-byte base64 key for ESPHome's native API (`api.encryption.key`). Generate with `openssl rand -base64 32`.
  </Card>
  <Card title="OTA password" icon="arrow-up-from-bracket">
    The password protecting over-the-air updates (`ota.password`). Prevents unauthorised firmware uploads.
  </Card>
  <Card title="External service credentials" icon="server">
    MQTT broker username/password, HTTP request API keys, and any other third-party credentials referenced in your config.
  </Card>
</CardGroup>

## Environment variable substitution in `secrets.yaml`

`secrets.yaml` is itself a YAML file loaded by ESPHome's custom loader, which supports the `!env_var` tag. This lets you pull values from environment variables at build time — useful for CI/CD pipelines where you don't want to check in any secrets at all.

```yaml
# secrets.yaml with environment variable references
wifi_ssid: !env_var WIFI_SSID
wifi_password: !env_var WIFI_PASSWORD
api_key: !env_var ESPHOME_API_KEY
ota_password: !env_var ESPHOME_OTA_PASSWORD
```

You can also provide a default value after the variable name (space-separated). If the environment variable is not set, the default is used:

```yaml
# Use MY_LOG_LEVEL env var, fall back to "INFO" if unset
log_level: !env_var MY_LOG_LEVEL INFO
```

If the environment variable is not set and no default is given, ESPHome raises a `yaml.MarkedYAMLError` pointing to the problematic line in `secrets.yaml`.

## Complete example

Here is a complete `secrets.yaml` and a device config that references it:

<CodeGroup>

```yaml secrets.yaml
# Wi-Fi
wifi_ssid: "My Home Network"
wifi_password: "correct-horse-battery-staple"
wifi_fallback_password: "fallback-access-1234"

# ESPHome native API
api_key: "dGhpcyBpcyBhIHRlc3Qga2V5IGZvciBkb2Nz"

# OTA
ota_password: "super-secret-ota-password"

# MQTT (if used)
mqtt_username: homeassistant
mqtt_password: "mqtt-broker-secret"
```

```yaml my-device.yaml
esphome:
  name: my-device
  friendly_name: My Device

esp32:
  board: esp32dev
  framework:
    type: arduino

wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password
  ap:
    ssid: "My Device Fallback"
    password: !secret wifi_fallback_password

api:
  encryption:
    key: !secret api_key

ota:
  - platform: esphome
    password: !secret ota_password

logger:
```

</CodeGroup>

## Using `esphome wizard`

If you are setting up ESPHome for the first time, the interactive wizard generates both your device YAML and a `secrets.yaml` stub:

```bash
esphome wizard my-new-device.yaml
```

The wizard prompts you for the device name, platform, Wi-Fi credentials, and OTA password, then writes:

- `my-new-device.yaml` — the device config using `!secret` references.
- `secrets.yaml` — populated with the values you entered, in the same directory.

## `.gitignore` best practices

`secrets.yaml` must **never** be committed to version control. Add it to your `.gitignore` immediately after creating it:

```text
# .gitignore
secrets.yaml
secrets.yml
```

<Warning>
If you accidentally commit `secrets.yaml`, treat all values in it as compromised. Rotate every credential — change your Wi-Fi password, regenerate the API key, reset the OTA password — before removing the file from git history with `git filter-repo` or `BFG Repo Cleaner`.
</Warning>

You can safely commit a `secrets.yaml.example` file that shows the expected keys with placeholder values, so team members and fresh checkouts know what to create:

```yaml
# secrets.yaml.example — copy to secrets.yaml and fill in real values
wifi_ssid: "YOUR_WIFI_SSID"
wifi_password: "YOUR_WIFI_PASSWORD"
wifi_fallback_password: "FALLBACK_AP_PASSWORD"
api_key: "BASE64_32_BYTE_KEY"
ota_password: "OTA_PASSWORD"
```

<Tip>
Generate a cryptographically random API key with:
```bash
openssl rand -base64 32
```
And a random OTA password with:
```bash
openssl rand -hex 16
```
</Tip>
